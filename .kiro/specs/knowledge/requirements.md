## 1. Executive Summary

我们为 **OpenClaw 龙虾各业务团队** 构建 **AI 知识库管理平台（Knowledge）**，解决 **团队业务知识分散、数据库元数据难以检索、提示词/Skill 无法共享复用** 的问题，预期实现 **团队知识资产的统一沉淀、语义化检索和高效共享，将知识查找效率提升 50% 以上**。

---

## 2. 问题陈述

### 2.1 谁有这个问题

OpenClaw 龙虾的多个业务团队成员，包括开发者、DBA、数据分析师、业务运营人员。

### 2.2 问题是什么

- **知识碎片化**：团队业务知识散落在个人笔记、聊天记录、文档中，新人入职或跨团队协作时无法快速获取
- **元数据不可搜索**：数据库表结构、字段含义、业务指标定义缺乏统一管理，开发者频繁重复询问 DBA
- **提示词重复造轮子**：各团队独立编写 AI 提示词/Skill，缺乏共享机制，优质提示词无法复用

### 2.3 为什么痛

> "每次新人来问表结构含义，我都要重新解释一遍，同样的问题回答了不下 20 次。" — DBA 团队成员
>
> "我知道其他团队有人写过类似的提示词，但我找不到，只能自己重新写一个。" — 业务运营人员

**数据支撑（预估）**：
- 团队成员平均每周花费 3-5 小时查找业务知识和元数据信息
- 跨团队知识复用率 < 10%
- 新人上手周期因知识获取困难延长 30%+

## 6. 方案概述

Knowledge 平台为用户提供一个统一的知识管理入口。用户可以将团队业务知识、数据库元数据、术语定义等内容录入系统，系统自动进行向量化处理。当用户需要查找信息时，只需输入自然语言描述，系统即可返回语义最相关的结果，无需记忆精确关键词。

平台同时提供 Skill 市场功能，团队成员可以将自己编写的优质提示词模板发布到市场，其他成员可以浏览、收藏和直接使用，避免重复造轮子。

团队管理者可以创建独立的知识空间，管理成员权限，确保敏感知识在团队内部隔离，同时通过仪表盘了解知识库的使用情况和团队贡献。

### 关键功能（5 个）

1. **知识条目管理** — 支持 Markdown 格式的知识创建、编辑、分类、标签管理
2. **向量语义搜索** — 基于自然语言的语义检索，支持多维度过滤
3. **数据库元数据管理** — MySQL 表结构自动导入、字段语义化、向量化搜索
4. **Skill/提示词市场** — 提示词模板的创建、发布、收藏、使用统计
5. **团队空间** — 多团队知识隔离、成员权限管理

## 8. 用户故事与需求

### Epic 假设

我们相信为 **OpenClaw 各团队成员** 添加 **统一的 AI 知识库平台** 将会 **大幅降低知识查找成本、提升跨团队知识复用率**，因为 **当前知识碎片化严重，语义搜索技术可以从根本上改变知识检索体验**。我们通过 **周均知识查找耗时** 在 **上线后 3 个月** 内验证。

### US-1: 知识条目 CRUD `P0`

**As a** 团队成员，**I want** 创建、编辑、查看和删除知识条目，**So that** 团队业务知识能够被结构化沉淀，不因人员流动而丢失。

**验收标准：**

```gherkin
Scenario: 创建知识条目
  Given 用户已登录并进入知识管理页面
  When 用户点击"新建知识"按钮
  Then 弹出创建表单，包含标题、内容（Markdown）、分类、标签字段

Scenario: 填写并保存知识条目
  Given 创建表单已打开
  When 用户填写标题、内容、选择分类和标签，点击保存
  Then 知识条目创建成功，显示成功提示
  And 条目出现在知识列表中
  And 内容自动进行向量化处理

Scenario: 创建时校验必填字段
  Given 创建表单已打开
  When 用户未填写标题直接点击保存
  Then 显示"标题不能为空"的错误提示，表单不提交

Scenario: 查看知识条目详情
  Given 知识列表中有条目
  When 用户点击某条目标题
  Then 进入详情页，显示完整的 Markdown 渲染内容、分类、标签、创建时间、更新时间

Scenario: 编辑知识条目
  Given 用户在知识详情页
  When 用户点击"编辑"按钮，修改内容并保存
  Then 条目内容更新成功，向量索引同步更新
  And 显示更新成功提示

Scenario: 删除知识条目
  Given 用户在知识列表或详情页
  When 用户点击"删除"按钮
  Then 弹出确认对话框，提示"确定删除该知识条目？"

Scenario: 确认删除
  Given 删除确认对话框已显示
  When 用户点击"确认"
  Then 条目被软删除，从列表和搜索结果中移除
  And 显示删除成功提示

Scenario: 取消删除
  Given 删除确认对话框已显示
  When 用户点击"取消"或按 Escape
  Then 对话框关闭，条目保持不变
```

**INVEST 检查：** ✅ 独立 | ✅ 可协商 | ✅ 有价值 | ✅ 可估算 | ✅ 足够小 | ✅ 可测试

### US-3: 向量语义搜索 `P0`

**As a** 用户，**I want** 通过自然语言描述搜索知识库，**So that** 无需记忆精确关键词即可快速找到相关信息。

**验收标准：**

```gherkin
Scenario: 基础语义搜索
  Given 知识库中已有向量化的知识条目
  When 用户在搜索框输入自然语言查询（如"订单表有哪些状态字段"）并提交
  Then 系统返回语义最相关的结果列表，按相关度排序
  And 搜索响应时间 < 2 秒

Scenario: 搜索结果展示
  Given 搜索返回了结果
  When 结果列表渲染完成
  Then 每条结果显示标题、内容摘要（高亮匹配片段）、来源类型（知识/元数据/Skill）、相关度分数

Scenario: 带过滤条件的搜索
  Given 用户在搜索页面
  When 用户输入查询并选择分类过滤条件
  Then 搜索结果同时满足语义相关性和过滤条件

Scenario: 点击搜索结果跳转
  Given 搜索结果列表已显示
  When 用户点击某条结果
  Then 跳转到对应的知识详情页/元数据详情页/Skill 详情页

Scenario: 无结果处理
  Given 用户输入了一个知识库中没有相关内容的查询
  When 搜索完成
  Then 显示友好的空状态提示："未找到相关结果，试试换个描述方式？"

Scenario: 空查询处理
  Given 用户在搜索页面
  When 用户未输入任何内容直接点击搜索
  Then 提示"请输入搜索内容"，不发起请求

Scenario: 搜索加载状态
  Given 用户提交了搜索请求
  When 等待后端响应期间
  Then 显示加载动画，搜索按钮禁用防止重复提交
```

**INVEST 检查：** ✅ 独立 | ✅ 可协商 | ✅ 有价值 | ✅ 可估算 | ✅ 足够小 | ✅ 可测试

### US-4: 数据库元数据导入 `P1`

**As a** DBA/管理员，**I want** 将 MySQL 数据库的表结构元数据导入系统，**So that** 团队成员可以通过平台查询表结构，减少重复询问。

**验收标准：**

```gherkin

  Given 系统已配置 MySQL 数据库连接（config.json）
  When 管理员在元数据页面点击"导入元数据"按钮
  Then 弹出确认对话框，显示将要连接的数据库名称

Scenario: 执行导入
  Given 用户确认导入
  When 系统连接数据库并提取元数据
  Then 自动提取所有表的表名、字段名、字段类型、注释等信息
  And 元数据自动向量化，可被语义搜索
  And 显示导入成功提示，包含导入的表数量

Scenario: 导入失败处理
  Given 数据库连接配置有误或数据库不可达
  When 系统尝试连接
  Then 显示明确的错误提示："数据库连接失败，请检查配置"

Scenario: 重复导入
  Given 元数据已导入过
  When 管理员再次触发导入
  Then 系统更新已有元数据（增量更新），不产生重复记录
  And 显示更新结果（新增 X 表，更新 Y 表）
```

**INVEST 检查：** ✅ 独立 | ✅ 可协商 | ✅ 有价值 | ✅ 可估算 | ✅ 足够小 | ✅ 可测试

## 12. 技术约束

> ⚠️ 技术栈选型由架构师确定，PRD 仅记录已确认的约束，不做技术决策。

### 12.1 前端

- 纯 HTML + TailwindCSS CDN + jQuery（不使用前端框架）
- CDN 依赖：tailwindcss、font-awesome 4.7、jQuery 3.7.1
- 设计风格、配色、字体由 `config.json` 统一配置，参见 `frontend-design.md`

### 12.2 后端

- Python FastAPI 框架
- MySQL 数据库（库名：`knowledge`）
- Redis 缓存和实时状态管理（端口 6381）
- RESTful API，支持 CORS 跨域

### 12.3 架构

- 前后端分离，独立目录（`knowledge/static/` + `knowledge/server/`）
- 后端服务端口：9999；前端服务端口：9998
- 后端脚本加载静态 HTML
- jQuery AJAX / Fetch 调用 API
- JS 模块拆分：`api.js`（接口层）+ `components.js`（通用组件）+ `app.js`（初始化）+ 各页面内联脚本

## 16. 风险与缓解（Pre-Mortem 分析）

**失败场景**："现在是 2026-09-27，Knowledge 平台上线 6 个月后彻底失败了。"

### 16.1 失败原因分析

| 类别 | 风险 | 可能性 | 影响 | 缓解方案 | 预警信号 |
|------|------|--------|------|---------|---------|
| 👥 人员 | 用户不愿主动贡献知识，平台成为空壳 | 高 | 高 | 设计激励机制（贡献排行榜）；降低录入门槛（支持快速创建）；管理层推动 | 上线 1 个月后周新增条目 < 10 |
| 👥 人员 | 关键开发人员离职导致项目无人维护 | 中 | 高 | 完善技术文档；代码规范化；确保至少 2 人熟悉全栈 | 核心开发者提出离职意向 |
| ⚙️ 流程 | 需求范围蔓延，各团队不断提新需求 | 高 | 中 | 严格按优先级排期；每个迭代只做 P0/P1；新需求走评审流程 | 单迭代需求数 > 计划的 150% |
| 💻 技术 | 向量搜索准确率不达预期，用户失去信任 | 中 | 高 | 先用小规模数据验证效果；支持关键词搜索作为降级方案；持续优化向量模型 | 搜索 Top5 命中率 < 50% |
| 💻 技术 | 元数据导入后表结构变更不同步 | 中 | 中 | 支持定时/手动重新导入；变更检测提醒 | 用户反馈元数据过时 > 3 次/周 |
| 💻 技术 | 前端纯 jQuery 方案在复杂交互场景下维护成本高 | 中 | 低 | 合理拆分 JS 模块；保持页面交互简洁；避免过度复杂的前端状态管理 | 单个 JS 文件超过 500 行 |
| 🌍 外部 | 竞品（如飞书知识库）推出类似功能，用户迁移 | 低 | 中 | 聚焦差异化功能（元数据管理、Skill 市场）；深度集成内部系统 | 竞品发布元数据管理功能 |

### 16.2 TOP 3 优先处理

1. **用户不贡献知识** → 上线前准备种子数据（至少 50 条高质量知识 + 全量元数据导入）；设置团队 KPI 关联
2. **向量搜索准确率** → 上线前用 100 条真实查询做 benchmark；准确率 < 60% 则延期上线
3. **需求范围蔓延** → 每个迭代开始前锁定需求清单，中途新增需求一律排到下个迭代

### 16.3 需要监控的预警信号

- □ 上线 2 周后日活用户 < 团队总人数的 20%
- □ 上线 1 个月后知识条目总数 < 100
- □ 搜索功能使用率 < 30%（说明用户不信任搜索结果）
