## 1. Executive Summary

我们为 **OpenClaw 龙虾各业务团队** 构建 **AI 知识库管理平台（Knowledge）**，解决 **团队业务知识分散、数据库元数据难以检索、提示词/Skill 无法共享复用** 的问题，预期实现 **团队知识资产的统一沉淀、语义化检索和高效共享，将知识查找效率提升 50% 以上**。

---

## 2. 问题陈述

### 2.1 谁有这个问题

OpenClaw 龙虾的多个业务团队成员，包括开发者、DBA、数据分析师、业务运营人员。

### 2.2 问题是什么

- **知识碎片化**：团队业务知识散落在个人笔记、聊天记录、文档中，新人入职或跨团队协作时无法快速获取
- **元数据不可搜索**：数据库表结构、字段含义、业务指标定义缺乏统一管理，开发者频繁重复询问 DBA
- **提示词重复造轮子**：各团队独立编写 AI 提示词/Skill，缺乏共享机制，优质提示词无法复用

### 2.3 为什么痛

> "每次新人来问表结构含义，我都要重新解释一遍，同样的问题回答了不下 20 次。" — DBA 团队成员
>
> "我知道其他团队有人写过类似的提示词，但我找不到，只能自己重新写一个。" — 业务运营人员

**数据支撑（预估）**：
- 团队成员平均每周花费 3-5 小时查找业务知识和元数据信息
- 跨团队知识复用率 < 10%
- 新人上手周期因知识获取困难延长 30%+

## 6. 方案概述

Knowledge 平台为用户提供一个统一的知识管理入口。用户可以将团队业务知识、数据库元数据、术语定义等内容录入系统，系统自动进行向量化处理。当用户需要查找信息时，只需输入自然语言描述，系统即可返回语义最相关的结果，无需记忆精确关键词。

平台同时提供 Skill 市场功能，团队成员可以将自己编写的优质提示词模板发布到市场，其他成员可以浏览、收藏和直接使用，避免重复造轮子。

团队管理者可以创建独立的知识空间，管理成员权限，确保敏感知识在团队内部隔离，同时通过仪表盘了解知识库的使用情况和团队贡献。

### 关键功能（5 个）

1. **知识条目管理** — 支持 Markdown 格式的知识创建、编辑、分类、标签管理
2. **向量语义搜索** — 基于自然语言的语义检索，支持多维度过滤
3. **数据库元数据管理** — MySQL 表结构自动导入、字段语义化、向量化搜索
4. **Skill/提示词市场** — 提示词模板的创建、发布、收藏、使用统计
5. **团队空间** — 多团队知识隔离、成员权限管理

## 8. 用户故事与需求

### Epic 假设

我们相信为 **OpenClaw 各团队成员** 添加 **统一的 AI 知识库平台** 将会 **大幅降低知识查找成本、提升跨团队知识复用率**，因为 **当前知识碎片化严重，语义搜索技术可以从根本上改变知识检索体验**。

---

### US-1: 知识条目管理 `P0`

**As a** 团队成员，**I want** 创建、编辑、分类和管理知识条目，**So that** 团队业务知识能够被结构化沉淀，不因人员流动而丢失。

**验收标准：**

```gherkin
Scenario: 创建知识条目
  Given 用户已登录并进入知识管理页面
  When 用户填写标题、内容（Markdown）、选择分类和标签，点击保存
  Then 知识条目创建成功，内容自动进行向量化处理
  And 条目出现在知识列表中，可被搜索到

Scenario: 编辑知识条目
  Given 用户拥有某知识条目的编辑权限
  When 用户修改内容并保存
  Then 条目内容更新，向量索引同步更新
  And 保留编辑历史记录

Scenario: 删除知识条目
  Given 用户拥有某知识条目的管理权限
  When 用户执行删除操作
  Then 条目被软删除（标记删除，不物理删除）
  And 条目从搜索结果中移除
```

**INVEST 检查：** ✅ 独立 | ✅ 可协商 | ✅ 有价值 | ✅ 可估算 | ✅ 足够小 | ✅ 可测试

---

### US-2: 向量语义搜索 `P0`

**As a** 用户，**I want** 通过自然语言描述搜索知识库，**So that** 无需记忆精确关键词即可快速找到相关信息。

**验收标准：**

```gherkin
Scenario: 语义搜索知识
  Given 知识库中已有向量化的知识条目
  When 用户输入自然语言查询（如"订单表有哪些状态字段"）
  Then 系统返回语义最相关的结果列表，按相关度排序
  And 搜索响应时间 < 2 秒

Scenario: 带过滤条件的搜索
  Given 用户在搜索页面
  When 用户输入查询并选择分类/标签过滤条件
  Then 搜索结果同时满足语义相关性和过滤条件

Scenario: 无结果处理
  Given 用户输入了一个知识库中没有相关内容的查询
  When 搜索完成
  Then 系统显示友好的无结果提示，并建议调整搜索词
```

**INVEST 检查：** ✅ 独立 | ✅ 可协商 | ✅ 有价值 | ✅ 可估算 | ✅ 足够小 | ✅ 可测试

### US-3: 数据库元数据管理 `P1`

**As a** DBA/开发者，**I want** 导入和浏览数据库表结构元数据，**So that** 团队成员可以通过语义搜索快速了解表结构和字段含义，减少重复询问。

**验收标准：**

```gherkin
Scenario: 导入 MySQL 表结构
  Given 系统已配置 MySQL 数据库连接
  When 管理员触发元数据导入
  Then 系统自动提取所有表的字段名、类型、注释等元数据
  And 元数据自动向量化，可被语义搜索

Scenario: 浏览表结构
  Given 元数据已导入
  When 用户进入元数据浏览页面
  Then 显示数据库表列表，点击可查看字段详情（字段名、类型、注释、索引）

Scenario: 搜索元数据
  Given 元数据已向量化
  When 用户搜索"用户手机号存在哪个表"
  Then 返回包含手机号相关字段的表和字段信息
```

**INVEST 检查：** ✅ 独立 | ✅ 可协商 | ✅ 有价值 | ✅ 可估算 | ✅ 足够小 | ✅ 可测试

### US-4: Skill/提示词市场 `P1`

```gherkin
  Given 用户拥有一个私有 Skill
  When 用户将 Skill 设置为公开
  Then Skill 出现在 Skill 市场中，其他用户可浏览和使用

Scenario: 收藏和使用 Skill
  Given 用户在 Skill 市场浏览
  When 用户收藏某个 Skill 并点击使用
  Then Skill 加入用户收藏列表
  And 使用次数统计 +1

Scenario: 按分类浏览 Skill
  Given Skill 市场中有多个分类的 Skill
  When 用户选择某个分类
  Then 仅显示该分类下的 Skill 列表
```

**INVEST 检查：** ✅ 独立 | ✅ 可协商 | ✅ 有价值 | ✅ 可估算 | ✅ 足够小 | ✅ 可测试

**As a** 团队管理者，**I want** 创建和管理团队知识空间，**So that** 不同团队的知识可以隔离管理，敏感信息不会泄露。

```gherkin
  Then 被邀请成员可按角色权限访问团队知识

Scenario: 知识隔离
  Given 存在多个团队空间
  When 用户搜索知识
  Then 仅返回用户有权限访问的团队空间中的结果
```

## 12. 技术约束

> ⚠️ 技术栈选型由架构师确定，PRD 仅记录已确认的约束，不做技术决策。

### 12.1 前端

- 纯 HTML + TailwindCSS CDN + jQuery（不使用前端框架）
- CDN 依赖：tailwindcss、font-awesome 4.7、jQuery 3.7.1
- 设计风格、配色、字体由 `config.json` 统一配置，参见 `frontend-design.md`

### 12.2 后端

- Python FastAPI 框架
- MySQL 数据库（库名：`knowledge`）
- Redis 缓存和实时状态管理（端口 6381）
- RESTful API，支持 CORS 跨域

### 12.3 架构

- 前后端分离，独立目录
- 后端服务端口：9999；前端服务端口：9998
- 后端脚本加载静态 HTML
- jQuery AJAX 调用 API
- JS 模块拆分：`api.js`（接口层）+ 各组件 JS

## 14. Out of Scope

| 不做的事 | 原因 |
|---------|------|
| 多语言国际化 | 当前用户均为中文用户，优先级不高 |
| 移动端适配 | 主要使用场景为桌面端办公，移动端需求不明确 |
| 实时协同编辑 | 技术复杂度高，当前阶段知识条目以单人编辑为主 |
| 自定义向量模型训练 | 使用通用向量模型即可满足需求，自训练 ROI 不高 |
| 第三方知识库导入（Notion/Confluence） | 未来考虑，当前聚焦核心功能 |
| 细粒度审计日志 | P3 需求，后续迭代考虑 |

## 16. 风险与缓解

| 风险 | 可能性 | 影响 | 缓解方案 |
|------|--------|------|---------|
| 向量搜索准确率不达预期 | 中 | 高 | 先用小规模数据验证效果；支持关键词搜索作为降级方案 |
| 用户不愿主动贡献知识 | 高 | 高 | 设计激励机制（贡献排行榜）；降低录入门槛（支持快速创建） |
| 元数据导入后表结构变更不同步 | 中 | 中 | 支持定时/手动重新导入；变更检测提醒 |
| 知识条目质量参差不齐 | 中 | 中 | 引入审核机制（P2）；支持用户反馈"内容过时" |
| 前端纯 jQuery 方案在复杂交互场景下维护成本高 | 中 | 低 | 合理拆分 JS 模块；保持页面交互简洁 |
